# Benchmarks

This document explains what `dirbase` should be compared with, what the benchmark suite measures, which dimensions are still pending data, and how to reproduce the results.

## Summary

The primary benchmark compares `dirbase` with `typicode/json-server`, because both tools turn JSON data into a local HTTP API with very little setup. The current automated script uses a deterministic six-resource dataset with 92,252 total rows and measures item lookups, filters, text search, sorting, pagination, and composite query workloads.

The benchmark scope is broader than read throughput alone. Every report should account for read paths, write paths, persistence correctness, startup behavior, memory, file watching, SSE latency, schema work, query correctness, and concurrent write safety. When a dimension has not been measured in a run, say `not measured` instead of omitting it.

Run the benchmark locally with:

```bash
scripts/benchmark_vs_json_server.sh
```

The latest generated reports are written to `benchmarks/results/benchmark-report-<timestamp>.md`, with raw `autocannon` JSON beside them.

## What To Compare

`json-server` is the closest baseline. It serves a single `db.json` or `db.json5` file as a fake REST API. `dirbase` supports that same single-file shape, but its main advantage is serving an existing folder where each `*.json` file is a resource.

`json-graphql-server` is useful for GraphQL-oriented comparisons. It generates a GraphQL API from a JSON data object. `dirbase` is broader rather than GraphQL-only: the same files are exposed through REST, read-only GraphQL, read-only SQL, and the overview UI.

Mockoon, Prism, WireMock, Postman Mock Servers, and MSW are adjacent API mocking tools. They are strongest when the goal is request matching, OpenAPI examples, proxying, response templating, or frontend test interception. They are not direct replacements for a folder-backed JSON datastore API.

`lowdb` is a JSON database library rather than a server. It is a good comparison for developer effort: with lowdb you still write the API surface yourself, while `dirbase` provides one from the filesystem.

PocketBase is a small backend, not a direct mock-server peer. It includes SQLite, auth, file storage, realtime subscriptions, and an admin UI. Compare it only when the question is whether a project needs a fuller local backend or a transparent JSON-folder API.

## Feature Comparison

| Capability | dirbase | json-server | Mockoon | Prism | lowdb |
| --- | --- | --- | --- | --- | --- |
| Folder of JSON files as API | Yes | No | Configured mocks | No | Library only |
| Single JSON database file | Yes | Yes | No | No | Yes |
| Persisted CRUD writes | Yes | Yes | Mock-oriented | No | Yes, through code |
| REST API | Yes | Yes | Yes | Yes | No built-in server |
| GraphQL API | Yes, read-only | No | No | No | No built-in server |
| SQL endpoint | Yes, read-only | No | No | No | No |
| File watching | Yes | Limited / version-dependent | No | No | No built-in server |
| Schema inference | Yes | No | No | OpenAPI-driven | No |
| Best fit | Local JSON datastore API | Simple fake REST API | Rich mock scenarios | Contract-first API mocking | Embedded JSON persistence |

## Current Scenarios

The automated benchmark currently exercises:

1. Item lookups on `tickets` and `projects`
2. Equality and range filters on `teams`, `projects`, and `deployments`
3. Text search on `tickets.summary`
4. Sorted and paginated collection reads on `members`, `tickets`, and `deployments`
5. Composite filter + sort + pagination workloads

The script uses equivalent server-specific query syntax where the two tools differ. For example, `dirbase` uses `risk_score:gte=70`, while `json-server` uses `risk_score_gte=70`.

## Benchmark Coverage

Each benchmark report should include this coverage matrix. The current script fills the read-heavy rows and leaves the remaining rows marked `not measured` until dedicated scenarios are implemented.

| Dimension | Current status | Required evidence |
| --- | --- | --- |
| Read latency and throughput | Measured | `autocannon` request rate, latency, errors, and non-2xx counts for item lookup, filter, text search, sort, pagination, and composite reads |
| Write latency | Not measured | `POST`, `PUT`, `PATCH`, and `DELETE` latency and throughput against writable resources |
| Persisted-write correctness | Not measured | Disk reads after mutation workloads proving the JSON files contain the expected rows and remain valid JSON |
| Cold start time | Not measured | Time from process spawn to ready endpoint success across small, medium, and large datasets |
| Memory usage | Not measured | Resident memory after startup and during sustained load for each dataset size |
| File watcher latency | Not measured | Time from external file add, edit, delete, or rename to updated HTTP responses |
| SSE event latency | Not measured | Time from filesystem change to `/events` notification delivery |
| Schema inference and export time | Not measured | Duration for schema inference, `/schema` export, and schema persistence on representative datasets |
| Query correctness | Not measured | Pairwise checks that equivalent `dirbase` and `json-server` requests return equivalent sorted payloads where feature parity exists |
| Concurrent write safety | Not measured | Parallel mutation workloads followed by JSON parse checks and expected row-count checks |

## Methodology

The benchmark script:

- builds `dirbase` in release mode
- starts `dirbase` and `json-server` locally on free ports
- waits until both servers are ready
- runs every scenario with and without warm-up
- repeats each scenario multiple times
- stores raw `autocannon` JSON for each run
- renders an aggregate Markdown report from the raw JSON

Default settings:

| Setting | Default |
| --- | ---: |
| Repeated runs | 3 |
| Duration per scenario | 10s |
| Connections | 50 |
| Warm-up duration | 3s |
| Warm-up connections | 1 |
| `json-server` version | `0.17.4` |

Prefer median values when discussing results. Avoid presenting best-run numbers as representative.

## Reproduce

From the repository root:

```bash
scripts/benchmark_vs_json_server.sh
```

For a fresh data rebuild:

```bash
FORCE_REBUILD_DATA=1 scripts/benchmark_vs_json_server.sh
```

For a longer run:

```bash
FORCE_REBUILD_DATA=1 RUNS=5 DURATION=15 CONNECTIONS=100 scripts/benchmark_vs_json_server.sh
```

The report paths are printed at the end of the run:

- `benchmarks/results/benchmark-summary-<timestamp>.json`
- `benchmarks/results/benchmark-report-<timestamp>.md`

## Presenting Results

Use a short README summary and keep detailed tables in generated benchmark reports. A good public claim has this shape:

```md
On a 92,252-row synthetic JSON dataset, `dirbase` was faster than `json-server`
across item lookup, filtering, text search, sorting, pagination, and composite
query scenarios in local `autocannon` runs. See `BENCHMARKS.md` and the latest
generated report for hardware, versions, exact scenarios, raw data, and caveats.
```

Keep the caveats explicit:

- results are local machine measurements, not universal performance guarantees
- benchmarked tools use different internals and query syntaxes
- benchmark reports must mark unmeasured dimensions as `not measured`
- `json-server` remains a good baseline for simple single-file fake REST APIs
- Mockoon, Prism, WireMock, Postman, MSW, lowdb, and PocketBase solve overlapping but different problems
