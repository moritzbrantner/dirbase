# Repository Rules

## Project Purpose

`dirbase` is a Rust CLI/API server that serves a directory or single file of JSON data as REST, GraphQL, SQL, and an embedded overview UI. Cargo builds use the checked-in `ui/dist/*` assets unless `DIRBASE_REBUILD_UI=1` is set.

## Key Directories

- `src/`: Rust server, CLI, storage, schema, query, GraphQL, SQL, and HTTP code. Keep behavior changes covered by Rust integration tests near the touched module.
- `ui/`: React overview UI built with Bun, Vitest, Playwright, Tailwind, and esbuild.
- `ui/dist/`: checked-in generated UI bundle consumed by Rust builds. Rebuild it only when UI source changes.
- `js/`: Bun package wrapper that bundles a JS launcher and copies release Rust binaries for npm publishing.
- `examples/`: small checked-in datasets for manual runs and demos.
- `benchmarks/`: benchmark fixtures, vendored comparison projects, and benchmark scripts. Generated benchmark work files belong under ignored benchmark output paths.
- `scripts/`: repository automation for tests, benchmark data, reports, and hygiene checks.
- `.github/workflows/`: CI definitions. Preserve existing job behavior unless the task explicitly changes CI.

## Commands

Use Bun 1.3.12 from `.bun-version` for JS/UI commands and the Rust toolchain from `rust-toolchain.toml`.

- Install UI dependencies: `bun install --cwd ui --frozen-lockfile`
- Install JS wrapper dependencies: `bun install --cwd js --frozen-lockfile`
- Dev server: `bun run dev` (runs `cargo run -- ./examples/school --bind 127.0.0.1:4444`)
- Fast tests: `bun run test`
- Full verification: `bun run verify`
- Lint: `bun run lint`
- Format: `bun run format`
- Format check: `bun run format:check`
- Rust production build: `bun run build`
- Rebuild checked-in UI bundle: `bun run build:ui`
- Build JS package wrapper: `bun run build:js`
- Repo hygiene report: `bun run hygiene`
- Full CI-style test script: `bash scripts/run_repo_tests.sh`

Do not add a root `release` or `publish` command unless the release process changes. The existing npm publish flow is tag/workflow driven by `.github/workflows/rust-to-bun.yml`.

## Files Agents Should Not Edit Manually

- `Cargo.lock`, `ui/bun.lock`, and `js/bun.lock`: update only through Cargo/Bun dependency commands.
- `ui/src/tailwind.generated.css`, `ui/dist/overview.css`, and `ui/dist/overview.js`: generated by `bun run build:ui`.
- `js/dist/` and `js/bin/`: ignored package build outputs generated by `bun run build:js` or `bun run --cwd js build`.
- `target/`, `node_modules/`, coverage folders, Playwright reports, and benchmark work directories: generated local artifacts.
- Benchmark vendored comparison projects under `benchmarks/json-server/` and `benchmarks/json-graphql-server/`: avoid changes unless the task is specifically about benchmarks.

## Search And Orientation

- Prefer `rg --files` for file discovery and `rg '<term>'` for text search.
- Use `cargo test -- --list` to inspect Rust test targets.
- Use `bun run --cwd ui test -- <pattern>` for focused Vitest runs.
- Use semantic search when available, for example: `semble search "schema inference foreign keys" --repo .`.

## Verification Notes

- Run targeted checks first for small changes, then `bun run verify` for broad or cross-stack changes.
- `scripts/run_repo_tests.sh` is intentionally expensive: it runs Rust tests, validates the test matrix, typechecks UI, runs UI unit tests, UI coverage, Playwright E2E, and JS wrapper tests.
- Playwright E2E may require Chromium installation: `cd ui && bunx playwright install --with-deps chromium`.
- Do not run `cd ui && bun test`; it uses Bun's test runner instead of Vitest and can produce false failures.
- Use `bun run hygiene` before handing off if generated files or local data may have been created.

## Coding Rules

- Apply clean code principles to every change: prefer clear names, small focused functions, low duplication, explicit data flow, and existing project patterns over new abstractions.
- Keep refactors behavior-preserving unless the user explicitly requests behavior changes.
- When duplicate logic appears across Rust, UI, or JS modules, extract a shared helper in the nearest appropriate module instead of editing copies independently.
- Run the relevant tests after changes. For broad repository changes, use `bun run verify` or `scripts/run_repo_tests.sh`.
